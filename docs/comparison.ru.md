# EndoCore vs FastAPI

Оба — современные Python ASGI-фреймворки. Они делают **разные ставки**. Здесь —
честное сравнение бок о бок, чтобы выбрать подходящий инструмент.

## Ключевое различие

| | EndoCore | FastAPI |
|---|---|---|
| **Роутинг** | **Файловый** — папка = путь, файл = метод | Декораторы (`@app.get(...)`) |
| **Источник истины** | Дерево каталога `Api/` | Python-код + декораторы |
| **Версионирование** | Первоклассное: папки `vN` сосуществуют | Вручную (роутеры/префиксы) |
| **ORM** | **Встроенная**, безопасная, sync + async | Нет (SQLAlchemy/Tortoise) |
| **Миграции** | **Встроенные** (`end migrate`, rollback, alter/rename) | Alembic (отдельно) |
| **CLI** | **Встроенный** (`end` — create/routes/check/migrate/openapi) | Нет (только uvicorn) |
| **Валидация** | Опциональный pydantic по параметру | pydantic везде (в ядре) |
| **DI** | `Depends` + провайдеры по типу/имени | `Depends` |
| **WebSockets** | Файловый `Socket.py` + pub/sub-менеджер | Декоратор `@app.websocket` |
| **Зависимости ядра** | 1 (`uvicorn`) | Starlette + pydantic + typing-extensions |
| **UI доков** | `/docs` (Swagger) встроенно | `/docs` + `/redoc` |

## Когда выбрать EndoCore

- Хотите, чтобы **структура и была API** — дерево это контракт, без расхождений.
- Хотите **версионирование как первоклассную, гарантированно изолированную**
  концепцию.
- Хотите **всё в комплекте**: ORM + миграции + кэш + DI + WebSockets + CLI в
  одном месте, с одной зависимостью в ядре.
- Любите **тонкие endpoint'ы + слой сервисов**, навязанные дизайном.
- Хотите небольшую **проверяемую** кодовую базу, которую можно прочитать целиком.

## Когда выбрать FastAPI

- Нужна **крупнейшая экосистема** и сообщество, проверено на масштабе.
- Полагаетесь на **pydantic везде** для моделей запрос/ответ и OpenAPI.
- Предпочитаете **явный роутинг декораторами** и готовы подключать свою ORM.
- Нужны фичи, которые EndoCore сознательно опускает (например, GraphQL через
  Strawberry, огромная экосистема плагинов).

## Одна задача, оба способа

**FastAPI**

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class UserIn(BaseModel):
    name: str

@app.post("/v1/user")
async def create_user(data: UserIn):
    return {"created": data.name}
```

**EndoCore** — маршрут *и есть* файл `Api/v1/User/Post.py`:

```python
from endocore import Request, Response
from pydantic import BaseModel

class UserIn(BaseModel):
    name: str

async def handler(request: Request, data: UserIn) -> Response:
    return Response.json({"created": data.name}, status=201)
```

Та же валидация, тот же OpenAPI. Разница — **где живёт маршрут**: в EndoCore URL
и метод — это расположение и имя файла, поэтому `end routes` всегда показывает
точную истину, а версионирование — это копия папки.

## Производительность

Примерно на равных — см. [Бенчмарки](benchmarks.md). На чистой диспетчеризации
FastAPI чуть быстрее на тривиальных статических маршрутах; EndoCore чуть быстрее
на динамических. В продакшене оба доминируются `uvicorn` и вашей БД.

## Честные ограничения EndoCore

- Меньше экосистема и сообщество (молодой сфокусированный проект).
- pydantic опционален, поэтому модели ответов по умолчанию не валидируются.
- Нет GraphQL и админ-панели (по дизайну).
- Нативные async-драйверы БД не используются; async-ORM выносит sync-ORM в
  тредпул (неблокирующе, работает для SQLite + PostgreSQL).

FastAPI отличный. EndoCore существует для тех, кому нужна **идея дерева папок** и
всё-в-комплекте со стеком на одной зависимости ядра.
