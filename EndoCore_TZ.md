# EndoCore — техническое задание

**Тип проекта:** file-based backend-фреймворк + CLI на чистом ASGI
**Язык:** Python
**Статус:** личный проект / спортивный интерес (не для рынка)
**Внешние зависимости ядра:** ровно одна — `uvicorn`

---

## 1. Суть проекта

EndoCore — backend-фреймворк, в котором **файловая структура является API-маршрутами**. Никаких ручных роутеров, декораторов и конфигов регистрации: положил файл в нужную папку — endpoint появился. CLI генерирует структуру, логирование встроено в ядро, тесты пишет пользователь сам.

Главная инженерная идея, вокруг которой держится весь проект: **и роутинг, и версионирование, и CLI — это операции над одним деревом папок.** Не три подсистемы, а одна абстракция (дерево) и разные виды операций над ней.

---

## 2. Философия и соглашения

1. **FILE = ROUTE.** Файл-обработчик = один endpoint.
2. **FOLDER = URL-сегмент.** `Api/User/Role/` → путь `/user/role`.
3. **FILE NAME = HTTP-метод.** `Post.py`, `Get.py`, `Patch.py`, `Delete.py`.
   - Имя метода нормализуется явно через `.upper()` в загрузчике (`Post` → `POST`), а не сравнивается посимвольно. Если кто-то создаст `post.py` руками — не сломается тихо.
   - На один метод — ровно один файл, поэтому коллизий регистра на case-insensitive ФС (macOS/Windows) не возникает.
4. **DYNAMIC SEGMENTS = `[id]`.** Папка `Api/User/[id]/` матчит `/user/42`, значение извлекается в параметры запроса.
5. **VERSION = первый сегмент пути вида `vЧИСЛО`.** Подробнее — раздел 5.

### Разделение по слоям (обязательно)

| Слой | Назначение |
|------|------------|
| **API** | endpoint-файлы. **Тонкие**: распарсил вход → позвал сервис → вернул ответ. |
| **Services** | вся бизнес-логика. Тяжёлый код живёт здесь. |
| **Models** | описание данных. |
| **Middleware** | обвязка запроса (auth, логирование). |
| **Utils** | чистые функции без побочных эффектов. |

> **Почему endpoint'ы обязаны быть тонкими:** это не эстетика, а функциональное требование версионирования (раздел 5). Если логика сидит в endpoint-файлах, а не в сервисах, каждая новая версия превращается в копипасту всей логики.

```

  auth_service.py
  permissions_service.py

Models/
  user.py
  role.py

Middleware/
  auth.py
  logger.py

Utils/
```

## 4. Механика фреймворка

### 4.1. Boot-процесс

1. Сканирует `/Api` через `pathlib` (`rglob("*.py")`).
2. Строит дерево маршрутов, определяя для каждого файла: версию, путь, метод, динамические сегменты.
3. Импортирует handler'ы через `importlib`.
4. Регистрирует endpoint'ы в реестре.

**Устойчивость к ошибкам импорта (обязательно):**
- Импорт каждого handler-файла обёрнут в `try/except`. Один сломанный файл **не роняет весь boot**.
- Ошибки собираются в список. В конце сканирования — сводка: `загружено N маршрутов, M файлов с ошибками` + пути + стектрейсы.
- Дерево маршрутов строится **один раз при старте** и кэшируется. В dev-режиме пересобирается по событию watcher'а, а не на каждый запрос.

### 4.2. Request flow

```
request → middleware → endpoint (API) → service → response
```

### 4.3. Резолвер маршрутов (сердце фреймворка)

Разбирает входящий путь `POST /v2/user/42/role` и находит файл `Api/v2/User/[id]/Role/Post.py`, извлекая `id=42`.

Правила матчинга:
- **Первый сегмент** проверяется по паттерну `^v\d+$` (`re`) → это версия.
- Остальные сегменты матчатся на имена папок; папка вида `[имя]` — динамический параметр.
- Имя файла (`Post`, `Get`, ...) сопоставляется с HTTP-методом после `.upper()`.

Это первое, что нужно написать физически: `app(scope, receive, send)` без резолвера не знает, какой файл звать.

## 5. Версионирование (путь «версия = папка»)

**Модель:** версия распространяется на **весь endpoint целиком со всеми методами**. Появилась `v2` — весь endpoint идёт на `v2`. При этом `v1` и `v2` **сосуществуют**, чтобы старые клиенты не ломались.

**Реализация:** версия — обычный первый сегмент URL и папки (`Api/v1/...`, `Api/v2/...`). Роутер не делает ничего особенного — матчит первый сегмент как версию по `^v\d+$`. **Версионирование не является отдельной подсистемой — это частный случай роутинга.** Кода «под версии» писать не нужно.

**Ключевое свойство:** после создания `v2` поведение `v1` не меняется ни на йоту. Если изменение в v2 может задеть v1 — версионирование фиктивно.

**Дефолтная версия (MVP):** запрос без префикса версии (`POST /user/role`) → **404**. Явное лучше неявного. (Опционально позже: резолв на максимальную версию, но обязательно с логированием, на какую версию ушёл запрос.)

**Изоляция сервисов при версии:**
- **Локальные сервисы** ресурса (`Api/vN/User/Services/`) — версионируемые, копируются при создании новой версии.
- **Глобальные сервисы** (`/Services/` в корне: `auth_service`, `permissions_service`) — **общие для всех версий**, не копируются.

Это сохраняет главное свойство: правка под v2 не задевает v1.

## 6. CLI-система (`end`)

### Генерация структуры

```
end create user/role post      # создаёт endpoint-файл метода POST + структуру
end create user/role           # создаёт базовую структуру endpoint'а
end dev                        # запускает сервер + watcher файлов
end test                       # опционально; тесты пишет пользователь сам
```

С учётом версий команда `create` должна знать целевую версию: либо явно (`end create v2/user/role post`), либо дефолт — «в последнюю существующую версию».

### Управление версиями

```
end version create 2           # копирует последнюю версию (v1) → v2:
                               #   дерево endpoint'ов + ЛОКАЛЬНЫЕ сервисы,
                               #   с телом файлов (не пустышки)
end version list               # показать существующие версии
end version create 2 --empty   # заготовки без тела (если реально нужно с нуля)
end version create 4 --from 1  # ответвиться от конкретной версии, а не от последней
```

**Физика `end version create`:** это `shutil.copytree` с фильтром (копируем endpoint'ы и локальные сервисы, пропускаем глобальные). Никакой магии — операция над деревом.

**Дефолт — копировать с телом**, потому что новая версия обычно «то же самое, но поменяю пару мест». `--empty` — опция, не поведение по умолчанию.

> Для MVP достаточно `end version create N` и `end version list`. Флаги `--empty` и `--from` — сахар на потом.

## 7. Система логирования

Не свой логгер с нуля — **обёртка над stdlib `logging`** (уровни, форматтеры, хендлеры уже есть) + middleware, которое замеряет время и маскирует чувствительные данные.

Каждый запрос логируется: endpoint, метод, payload, response, время выполнения, стектрейс ошибки.

```
[INFO]  POST /v2/user/role 12ms
[ERROR] validation failed in RoleService
```

### Маскирование чувствительных данных (обязательно с первого дня)

Логирующее middleware перехватывает payload **на входе**, до любого шифрования. Значит в лог полетит сырой JSON с паролями в открытом виде — независимо от того, как данные потом лягут в БД (шифрование at rest эту проблему не решает).

Решение — маскирование **на слое логгера**: список чувствительных ключей (`password`, `token`, `authorization`, `secret`, ...) и замена значений на `***` перед записью. Две строки кода, но заложить нужно сразу.

## 8. Технологический стек (чистый ASGI)

### Что берётся извне — ровно одно

```
uvicorn        # ASGI-сервер: открывает сокет, парсит HTTP в scope,
               # вызывает app(scope, receive, send). Даёт --reload из коробки.
```

«С нуля» = писать резолвер, загрузчик, Request/Response, middleware самому. «С нуля» ≠ писать TCP-сервер и HTTP-парсер — это другая планета (`h11`/`httptools`), к идее фреймворка отношения не имеет. Uvicorn — правильная граница «с нуля».

### Ядро — из стандартной библиотеки (ставить не надо)

| Модуль | Для чего |
|--------|----------|
| `pathlib` | обход дерева `/Api`, построение маршрутов (`rglob`) |
| `importlib` | динамический импорт handler-файлов (тот самый «автоскан») |
| `re` | матчинг версии `^v\d+$` и динамических `[id]` |
| `inspect` | чтение сигнатур handler'ов, наличие `init()` |
| `logging` | транспорт логов (форматтеры, уровни, хендлеры) |
| `json` | тело запроса/ответа |
| `time` | замер времени выполнения |
| `traceback` | стектрейсы в лог |
| `dataclasses` | описание маршрута как структуры (`@dataclass Route`) |
| `urllib.parse` | разбор query-строки (`parse_qs`), декодирование URL |
| `asyncio` | работа с корутинами ASGI |
| `argparse` | CLI-команды `end` |

Весь резолвер, загрузчик, реестр, обёртка логгера и CLI пишутся **без единой сторонней зависимости** сверх `uvicorn`.

### Что делает чистый ASGI лично на тебе

ASGI-приложение — это одна async-функция `async def app(scope, receive, send)`:
- **`scope`** — dict с метаданными (метод, путь, заголовки).
- **`receive`** — корутина, из которой достаёшь тело запроса чанками.
- **`send`** — корутина, в которую пишешь ответ.

Три вещи, которые раньше давал бы Starlette, а теперь пишутся руками:
1. **Свои `Request` / `Response`** — пара dataclass'ов. `Request` читает `scope` и собирает тело через `receive`; `Response` формирует `send`.
2. **Разбор заголовков** — в ASGI приходят как список **байтовых** кортежей `[(b"content-type", b"application/json")]`, не строк. Всё в `bytes`, декодируешь сам. Тут легко споткнуться.
3. **Middleware-цепочка** — на старте проще всего список функций, каждая получает `Request` и либо возвращает ранний `Response` (auth), либо пропускает дальше. Позже можно перейти на классический паттерн «обёртка вокруг app».

**Упрощение для MVP:** тело через `receive` приходит потоком (чанки с `more_body`). Для JSON-API собираем всё тело в память целиком, потом парсим. Стриминг (большие файлы) существует, но на старте игнорируется сознательно — не строить сразу под него.

### Добавить позже, по потребности (не в MVP)

| Библиотека | Когда |
|-----------|-------|
| `watchfiles` | свой watcher для `end dev`; на старте живём на `uvicorn --reload` |
| `pydantic` | валидация входных данных и моделей; сначала — сырые dict |
| `pytest` | зависимость пользователя (его тесты), не фреймворка |

## 9. Объём MVP

**Входит:**
- file-based роутинг (папка = сегмент, файл = метод);
- динамические маршруты `[id]`;
- версионирование через папку-версию `vN` (первый сегмент, `^v\d+$`);
- GET/POST handler'ы;
- слой сервисов (тонкие endpoint'ы);
- middleware-система;
- CLI: `end create`, `end dev`, `end version create N`, `end version list`;
- логирование с маскированием чувствительных полей;
- папка `Tests/` под ручные тесты пользователя.

**Точка старта разработки:** резолвер маршрутов (`Core/router.py`). Он определяет жизнеспособность всей идеи, на нём завязаны `[id]`, версии и middleware-скоупинг. ~80 строк на `pathlib` + `re` + `dataclasses`, ноль внешних зависимостей. После него — `Request`/`Response`, запуск под uvicorn, дальше CLI и логирование.

**Не входит в MVP (сознательно отложено):**
- дефолтная версия как алиас на последнюю (пока 404);
- флаги `--empty`, `--from` у `version create`;
- свой watcher (`watchfiles`);
- `pydantic`-валидация;
- стриминг тела запроса.

## 10. Зафиксированные решения (чтобы не пересматривать через полгода)

1. **Версии — папки, не метаданные.** Поле `version` внутри файла отвергнуто: раз версия на весь endpoint, её место — папка `vN`, а не ярлык в файле. Всё видно глазами в дереве.
2. **Endpoint'ы тонкие, логика в сервисах.** Это условие того, чтобы версионирование не стало копипастой.
3. **Локальные сервисы версионируются, глобальные — общие.**
4. **Маскирование логов на слое логгера, до шифрования.** Шифрование в БД проблему логов не закрывает.
5. **Чистый ASGI, единственная внешняя зависимость — uvicorn.** Всё, что составляет идею (магия дерева), пишется самому из stdlib; всё, что идею не составляет (HTTP-транспорт), берётся готовым.
6. **Boot не падает от одного сломанного файла** — сводка ошибок вместо глухого `ImportError`.
